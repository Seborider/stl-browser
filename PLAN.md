# STL Browser ŌĆö Implementation Plan

## Context

Greenfield macOS desktop app: a "Photos.app for STLs" that shows a scrollable grid of thumbnails rendered from the user's 3D model files (STL, 3MF, OBJ). Built with Tauri 2 (Rust backend + React/TS frontend). The user is a TypeScript developer new to Rust; explanations of Rust-specific patterns appear inline.

Repo is empty (only a "first commit"). This document is the architecture and phased build plan; no code is written yet. Update in place as the project evolves.

**Why this layout**
- `src/` and `src-tauri/` are Tauri convention; don't fight it.
- Rust is split by **responsibility** (db, scan, mesh, cache, ipc), not by layer. Each module owns its data and exposes a narrow API to the rest.
- `state.rs` centralizes the one `AppState` that Tauri commands receive ŌĆö standard Tauri pattern.
- `generated/bindings.ts` is checked in but auto-regenerated; treat it like a build artifact you happen to commit for DX.
- Frontend `thumbs/` is its own folder because the queue + worker + protocol form a cohesive subsystem that's the trickiest code in the app.

## 2. Phased build order

**Guiding rule:** every phase ends with something you can run and click. Thumbnails are the risky bit and come late, after the shell and data layer are both real.

### Phase 0 ŌĆö Tooling (┬Į day)
- Install Rust via rustup, verify `cargo --version`.
- Install Node 20 LTS + pnpm.
- `pnpm create tauri-app` ŌåÆ pick React + TypeScript + Vite.
- Verify `pnpm tauri dev` opens a window.
- Add Tailwind + shadcn init.
- **Exit criterion:** blank window with one Tailwind-styled button.

### Phase 1 ŌĆö Shell UI with dummy data (1ŌĆō2 days)
- Three-pane layout (sidebar / grid / inspector) with resize handles.
- Hard-code 2 libraries and ~2000 mock files in a TS fixture.
- Wire up react-virtuoso grid with solid-color placeholder tiles.
- Grid size slider (Finder-style).
- Sort dropdown, search box (client-side fuzzy, no real data yet).
- Inspector renders mock metadata.
- Keyboard nav (arrows, enter).
- **Exit criterion:** 2000 fake files scroll at 60fps. Everything is fake but the UI is done.

### Phase 2 ŌĆö Rust data layer (2ŌĆō3 days)
- SQLite init (bundled), migrations runner.
- `AppState` holding a `rusqlite::Connection` behind `Arc<Mutex<_>>`.
- Implement libraries CRUD + `list_libraries`, `add_library`, `remove_library` commands.
- Implement `list_files(library_id, sort, search) -> Vec<FileEntry>`.
- Set up ts-rs type generation; wire into `cargo build` via `build.rs`.
- Replace frontend fixtures with real IPC calls. No files exist yet, so the grid will be empty.
- **Exit criterion:** you can add/remove a library via the UI and it persists across app restarts.

### Phase 3 ŌĆö Scanning + metadata (2ŌĆō3 days)
- `walkdir` recursive scan, filter by extension.
- Hash each file (blake3 of path+mtime+size) ŌåÆ `cache_key`.
- Insert `files` rows; emit `files:added` in batches of ~100.
- Spawn a tokio task per library that parses mesh metadata (bbox, tri count, volume, area). Emit `metadata:ready` per file.
- **Exit criterion:** add a real folder of STLs and see them appear in the grid with real metadata and no thumbnails.
- **Phase 3 ŌĆö Status: complete.**

### Phase 4 ŌĆö File watching (1 day)
- `notify-debouncer-full` wrapping `notify::RecommendedWatcher`.
- On event: diff against DB, emit `files:added` / `files:removed` / `files:updated`.
- Re-scan metadata for updated files.
- **Exit criterion:** dropping a new STL into a watched folder makes it appear within ~1s.
- **Phase 4 ŌĆö Status: complete.**

### Phase 5 ŌĆö Thumbnail pipeline (3ŌĆō5 days ŌĆö THE RISKY PART)
- **Do the three spikes in ┬¦7 first.** Merge them into `src-tauri/` and `src/thumbs/` only once proven.
- Build the real pipeline:
  - Rust emits `thumbnails-needed` with cache keys that have no row in `thumbnails`.
  - Renderer priority queue (visible-first).
  - Web Worker fetches mesh via asset protocol, renders via three.js + OffscreenCanvas, posts PNG bytes to main thread.
  - Main thread calls `save_thumbnail` (raw-body IPC).
  - Rust writes `<cache_dir>/thumbnails/<hash>.png` and inserts thumbnail row; emits `thumbnail-ready`.
  - Grid swaps placeholder for real image.
- Integrate with grid scroll position: bump priority for visible-and-near tiles.
- **Exit criterion:** 500-file folder fully thumbnailed within ~1 min, cache persists across restarts, scrolling stays smooth throughout.
- **Phase 5 ŌĆö Status: complete.**

### Phase 6 ŌĆö Detail viewer (1ŌĆō2 days)

## 3. IPC contract

All types generated from Rust via ts-rs into `src/generated/bindings.ts`. Below is the human-readable contract.

### Commands (renderer ŌåÆ Rust, via `invoke`)

| Name | Rust signature | TS signature | Purpose |
|------|----------------|--------------|---------|
| `list_libraries` | `() -> Result<Vec<Library>, IpcError>` | `(): Promise<Library[]>` | Sidebar population |
| `add_library` | `(path: String) -> Result<Library, IpcError>` | `(path: string): Promise<Library>` | User picked a folder; start scan |
| `remove_library` | `(id: i64) -> Result<(), IpcError>` | `(id: number): Promise<void>` | Also deletes child files from DB |
| `rescan_library` | `(id: i64) -> Result<(), IpcError>` | `(id: number): Promise<void>` | Manual refresh |
| `list_files` | `(query: FileQuery) -> Result<Vec<FileEntry>, IpcError>` | `(q: FileQuery): Promise<FileEntry[]>` | Grid population; supports library_id, sort, search |
| `get_file_details` | `(id: i64) -> Result<FileDetails, IpcError>` | `(id: number): Promise<FileDetails>` | Inspector + viewer; includes full metadata |
| `get_mesh_asset_url` | `(id: i64) -> Result<String, IpcError>` | `(id: number): Promise<string>` | Returns `asset://...` URL for the worker/viewer to fetch |
| `save_thumbnail` | `(cache_key: String, png: Vec<u8>) -> Result<(), IpcError>` (raw-body) | `(key: string, bytes: Uint8Array): Promise<void>` | Persist worker output |
| `get_thumbnail_url` | `(cache_key: String) -> Result<Option<String>, IpcError>` | `(key: string): Promise<string \| null>` | Asset URL for an `<img>` tag |
| `reveal_in_finder` | `(path: String) -> Result<(), IpcError>` | `(path: string): Promise<void>` | Shell out to `open -R` |
| `get_settings` | `() -> Result<Settings, IpcError>` | `(): Promise<Settings>` | Theme, grid size, print bed, units |
| `set_settings` | `(patch: SettingsPatch) -> Result<Settings, IpcError>` | `(patch: SettingsPatch): Promise<Settings>` | Merge-write |

### Events (Rust ŌåÆ renderer, via `emit`)

| Name | Payload | When emitted |
|------|---------|--------------|
| `scan:started` | `{ libraryId: number }` | A library scan begins |
| `scan:progress` | `{ libraryId, scanned, estimated }` | Every ~250ms during scan |
| `scan:completed` | `{ libraryId, total }` | Scan finished |
| `files:added` | `{ files: FileEntry[] }` | Scanner found new files (batched) |
| `files:removed` | `{ fileIds: number[] }` | Watcher detected deletions |
| `files:updated` | `{ files: FileEntry[] }` | Watcher detected mtime changes |
| `metadata:ready` | `{ fileId, metadata }` | Async mesh parse finished |
| `thumbnails:needed` | `{ items: { fileId, cacheKey, assetUrl }[] }` | After scan/update, files lacking valid thumbs |
| `thumbnails:ready` | `{ cacheKey, url }` | After worker+Rust persisted a thumbnail |
| `scan:error` | `{ libraryId, message }` | Permission / IO failure |

**Notes**
- All events prefixed with their namespace (`scan:`, `files:`, `thumbnails:`) ŌĆö Tauri allows this, the colon is just a string convention.
- `save_thumbnail` uses Tauri 2's raw-body IPC ŌĆö `invoke(cmd, args, { headers: ..., body: ArrayBuffer })`. Exact API confirmed in Spike 3. Rationale: a JSON-encoded 200KB PNG is ~270KB of UTF-8 and costs a JSON.parse on both sides; raw-body is zero-copy.

## 4. SQLite schema

```sql
-- Stored at: $APP_DATA/library.db
-- Migrations live in src-tauri/src/db/migrations.rs, applied on startup.

PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;

CREATE TABLE libraries (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  path        TEXT    NOT NULL UNIQUE,    -- absolute path
  name        TEXT    NOT NULL,           -- user-visible; defaults to basename
  added_at    INTEGER NOT NULL            -- unix millis
);

CREATE TABLE files (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  library_id   INTEGER NOT NULL REFERENCES libraries(id) ON DELETE CASCADE,
  rel_path     TEXT    NOT NULL,          -- relative to library.path, forward slashes
  name         TEXT    NOT NULL,          -- basename, for sort/search
  extension    TEXT    NOT NULL,          -- lowercase: stl, 3mf, obj
  size_bytes   INTEGER NOT NULL,
  mtime_ms     INTEGER NOT NULL,
  scanned_at   INTEGER NOT NULL,
  cache_key    TEXT    NOT NULL,          -- blake3(path + mtime + size), hex
  UNIQUE(library_id, rel_path)
);
CREATE INDEX idx_files_library      ON files(library_id);
CREATE INDEX idx_files_name         ON files(name COLLATE NOCASE);
CREATE INDEX idx_files_mtime        ON files(mtime_ms);
CREATE INDEX idx_files_size         ON files(size_bytes);
CREATE INDEX idx_files_cache_key    ON files(cache_key);

CREATE TABLE mesh_metadata (
  file_id         INTEGER PRIMARY KEY REFERENCES files(id) ON DELETE CASCADE,
  bbox_min_x REAL, bbox_min_y REAL, bbox_min_z REAL,    -- nullable on parse_error rows
  bbox_max_x REAL, bbox_max_y REAL, bbox_max_z REAL,    -- "
  volume_mm3      REAL,                    -- nullable: Some only when edge-watertight
  surface_area_mm2 REAL,
  triangle_count  INTEGER,                 -- nullable on parse_error rows
  computed_at     INTEGER NOT NULL,
  parse_error     TEXT,                    -- non-null if parse failed; row still exists so we don't retry
  CHECK (parse_error IS NOT NULL
         OR (bbox_min_x IS NOT NULL AND bbox_min_y IS NOT NULL AND bbox_min_z IS NOT NULL
             AND bbox_max_x IS NOT NULL AND bbox_max_y IS NOT NULL AND bbox_max_z IS NOT NULL
             AND triangle_count IS NOT NULL))
);

CREATE TABLE thumbnails (
  cache_key     TEXT PRIMARY KEY,          -- matches files.cache_key
  generated_at  INTEGER NOT NULL,
  width         INTEGER NOT NULL,
  height        INTEGER NOT NULL
);
CREATE INDEX idx_thumbnails_generated ON thumbnails(generated_at);

CREATE TABLE settings (
  key    TEXT PRIMARY KEY,
  value  TEXT NOT NULL                    -- JSON
);

CREATE TABLE schema_version (version INTEGER PRIMARY KEY);
```

**Design notes**
- **Thumbnails keyed by `cache_key`, not `file_id`** ŌĆö if two identical files exist across libraries, the thumbnail dedupes. The PNG itself lives at `<cache_dir>/thumbnails/<cache_key>.png`. A periodic GC sweep removes PNGs without a matching `cache_key` in `files`.
- **`parse_error` stored in `mesh_metadata`** ŌĆö lets us avoid re-parsing broken files every launch. If the file's mtime changes the row gets replaced.
- **bbox_* and triangle_count nullable** ŌĆö a file with a `parse_error` has no geometry. A CHECK constraint enforces "either parse_error OR full geometry", so non-null geometry always round-trips through `get_file_details`.
- **`rel_path` instead of absolute** ŌĆö moving a library on disk only requires updating `libraries.path`.
- **`COLLATE NOCASE` on name index** ŌĆö case-insensitive sort matches Finder's default.
- **WAL mode** ŌĆö concurrent scanner writes + UI reads without blocking.

### Queue design (renderer)

- A single priority queue keyed on `cacheKey`, with priority tiers:
  1. **Visible tiles** (highest) ŌĆö derived from virtuoso's rangeChanged.
  2. **Near-visible** (┬▒1 viewport) ŌĆö prefetch.
  3. **Rest of current library**.
  4. **Other libraries** (lowest).
- Re-prioritize on scroll with a 100ms debounce.
- De-dup: never queue the same key twice.
- Worker concurrency: **1 worker to start** (a single WebGL context is cheaper than 4). Revisit if throughput is insufficient; the interface supports N workers trivially.
- In-flight limit: at most 1 job per worker at a time. Queue rest.

### Web Worker internals (`render-worker.ts`)

1. Receive `{ cacheKey, assetUrl, fileExt }` from main thread.
2. `fetch(assetUrl)` ŌåÆ `arrayBuffer()` ŌĆö works inside workers in Tauri's webview as long as asset protocol scope allows the library folder.
3. Parse via three.js loader (`STLLoader`, `ThreeMFLoader`, `OBJLoader`).
4. Geometry downsample: if triangle count > 200k, use `BufferGeometryUtils.mergeVertices` and/or simplifier; for v1, just cap normal computation cost and render at lower canvas size if needed.
5. Render on a single reused `OffscreenCanvas` (e.g. 512├Ś512):
   - `PerspectiveCamera` positioned to fit the mesh's bounding sphere.
   - Two directional lights + ambient; standard material, light-gray.
   - One render call.
6. `canvas.convertToBlob({ type: 'image/png' })` ŌåÆ `arrayBuffer()`.
7. `postMessage({ cacheKey, pngBuffer }, [pngBuffer])` ŌĆö transferable, zero-copy.
8. Free geometry/material, move to next job.

### IPC transfer (Renderer ŌåÆ Rust)

- Main thread receives the transferable ArrayBuffer.
- Calls `invoke('save_thumbnail', { cacheKey }, { body: pngBuffer })` using Tauri 2 raw-body IPC.
- Rust handler writes to `<cache_dir>/thumbnails/<cacheKey>.png` (atomic: write to `.tmp`, then `rename`), inserts a row in `thumbnails`, and emits `thumbnails:ready`.
- **Fallback** if raw-body proves unreliable: write PNG from main thread to a temp file via `tauri-plugin-fs`, pass the temp path to `save_thumbnail` which moves it into place. Decided during Spike 3.

### Staleness / invalidation

- Cache key = `blake3(abs_path + mtime_ms + size_bytes)` ŌĆö deterministic and cheap.
- On file update (notify): regenerate `cache_key`, replace `files` row, emit `thumbnails:needed` for the new key. Old PNG becomes orphaned.
- **GC sweep** (background, on launch + every ~24h of runtime): list all PNGs under `thumbnails/`, delete any whose name isn't in `files.cache_key`. Safe because `files` always has a row before `thumbnails`.
- On file deletion: trigger cascades remove `mesh_metadata`; the GC sweep cleans the PNG + `thumbnails` row.

### Back-pressure

If the queue explodes (e.g. a 50k-file library just got added), cap enqueued jobs at a rolling window (e.g. 2000) and let Rust re-emit `thumbnails:needed` as work drains. This prevents unbounded memory use.

## 7. Risks and spikes (do these first)

Each spike is a standalone throwaway project, ~┬Į day each, that proves the risky assumption before you commit to the architecture. Do them in this order during Phase 5 prep.

### Spike 1: OffscreenCanvas + three.js in a Web Worker

- Plain HTML page, no React, no Tauri.
- Worker loads an embedded STL URL, renders to 512├Ś512 OffscreenCanvas, posts a PNG blob back.
- Measure: time from message-in to bytes-out for 100k-triangle STL and 1M-triangle STL.
- **Success:** <300ms for 100k tris, <2s for 1M tris.
- **What could kill it:** OffscreenCanvas WebGL support in the Tauri webview (WKWebView on macOS) ŌĆö confirm it works. If it doesn't, fall back to a hidden main-thread canvas and use `requestIdleCallback`; worse but still viable.

### Spike 2: Tauri asset protocol for large files

- Minimal Tauri app, renderer `fetch('asset://localhost/.../big.stl')` for a 100MB file.
- Verify it streams (doesn't blow heap) and scope config works.
- Measure throughput.
- **Success:** 100MB STL reaches the renderer in <2s without OOM.
- **Fallback:** custom protocol handler with manual streaming ŌĆö Tauri 2 supports `tauri::uri_scheme_protocol`.

### Spike 3: Binary IPC throughput (renderer ŌåÆ Rust)

- Send a 1MB `Uint8Array` to a Rust command.
- Try raw-body IPC first; verify Tauri 2 API shape.
- Compare against JSON-encoded `Vec<u8>` (expect ~4├Ś slower and higher CPU).
- **Success:** <20ms round-trip for 1MB.
- **Fallback:** write to temp file, pass path.

### Spike 4: `notify` on a large directory

- Watch a folder with 10k files on macOS.
- Create, modify, delete in bursts; measure event latency and CPU.
- Verify debouncer correctly coalesces rapid changes (e.g. `unzip` of a 500-file archive).
- **Success:** <500ms from mutation to event; <5% sustained CPU in steady state.
- FSEvents already coalesces; the debouncer is belt + braces.

### Other risks (no spike needed, just be aware)

- **Volume calculation for non-watertight meshes** ŌĆö not meaningful; store NULL and show "ŌĆö" in UI.
- **3MF with multiple objects per file** ŌĆö for v1, treat as a single mesh (merge components); revisit for v2.
- **Unicode filenames** ŌĆö `PRAGMA encoding = 'UTF-8'` already default; don't normalize, store what macOS returns.
- **Symlinks in scan** ŌĆö follow by default but detect cycles (walkdir handles this).

## 8. macOS specifics

### Distribution target

**Developer ID Application** distribution outside the Mac App Store (confirmed). This avoids full sandboxing and simplifies folder access. If we ever want MAS later, security-scoped bookmarks for persistent folder permissions are a non-trivial addition.

### Signing + notarization

- Join Apple Developer Program ($99/yr).
- Create a **Developer ID Application** certificate in Keychain via Xcode or the web portal.
- Find your Team ID (Apple Developer account ŌåÆ Membership).
- Create an app-specific password for `notarytool` in appleid.apple.com.
- Store credentials in the macOS Keychain via:
  ```
  xcrun notarytool store-credentials "AC_PASSWORD" \
    --apple-id "<your-apple-id>" \
    --team-id "<team-id>" \
    --password "<app-specific-password>"
  ```
- Configure `tauri.conf.json`:
  ```json
  "bundle": {
    "macOS": {
      "signingIdentity": "Developer ID Application: Your Name (TEAMID)",
      "providerShortName": "TEAMID",
      "minimumSystemVersion": "12.0",
      "entitlements": "entitlements.plist",
      "hardenedRuntime": true
    }
  }
  ```
- Tauri 2's bundler calls `codesign` + staples the notarization ticket. You can also invoke `notarytool submit --wait` manually after `cargo tauri build`.

## 9. Rust onboarding (for this project specifically)

### Concepts you WILL meet and need

1. **Ownership & borrowing basics**
   - `&str` (borrowed string slice) vs `String` (owned). IPC payloads come in as `String`; pass `&str` where possible.
   - `.clone()` when you need to keep something around in a closure ŌĆö cheap for small data, fine for paths. Don't prematurely optimize.

2. **`Result<T, E>` and `?`**
   - Every fallible function returns `Result`. `?` propagates the error upward.
   - Tauri commands return `Result<T, E>` where `E: Serialize` ŌåÆ becomes a rejected Promise on the TS side.

3. **`Option<T>`**
   - Rust's `null`. Use `.unwrap_or(default)`, `.map(|v| ...)`, `if let Some(v) = ...`.

4. **Async with `tokio`**
   - `async fn` returns a future; `.await` runs it.
   - `tokio::spawn` for background tasks (scanner, watcher).
   - Tauri commands can be `async fn`; the macro handles it.

5. **Shared state: `Arc<Mutex<T>>` / `Arc<RwLock<T>>`**
   - `Arc` = reference-counted pointer (TS equivalent: a JS object held by reference).
   - `Mutex` = exclusive lock; `RwLock` = many readers OR one writer.
   - Standard pattern: `Arc<Mutex<Connection>>` for the DB, stored in `AppState`.
   - Rule of thumb: acquire the lock, do the minimum work, drop it. Never hold across `.await`.

6. **Derive macros**
   - `#[derive(Serialize, Deserialize, Debug, Clone, TS)]` on structs that cross the IPC boundary.
   - `#[ts(export)]` on structs you want in generated bindings.

7. **Tauri command macro**
   - `#[tauri::command] async fn foo(state: State<'_, AppState>, arg: String) -> Result<Foo, IpcError> { ... }`
   - Registered in `main.rs` via `tauri::generate_handler![foo, bar, ...]`.

8. **Error modeling with `thiserror`**
   ```rust
   #[derive(thiserror::Error, Debug, Serialize)]
   pub enum IpcError {
     #[error("not found: {0}")]
     NotFound(String),
     #[error(transparent)]
     Io(#[from] std::io::Error),
     #[error(transparent)]
     Sqlite(#[from] rusqlite::Error),
   }
   ```

### Concepts you can IGNORE for v1

- **Lifetimes with explicit annotations** (`'a`). The compiler infers almost all of them for this project. When the compiler demands one, it tells you where and why.
- **Unsafe** ŌĆö zero need.
- **Writing your own traits / generics** ŌĆö consuming trait bounds from libraries is fine; don't invent your own.
- **Pinning, `Pin<Box<...>>`** ŌĆö tokio abstracts it.
- **Macros by example (`macro_rules!`)** ŌĆö you'll USE macros but won't write them.
- **Send / Sync bounds** ŌĆö tokio + Tauri surface these if you mess up, and the error messages are actually readable.

### Where you will struggle

- **Borrow checker fights when you try to modify data from inside a closure.** Solution: clone into the closure, or restructure.
- **Async + Mutex across `.await` points.** Use `tokio::sync::Mutex` (async-aware) rather than `std::sync::Mutex` when the lock must be held across `.await`. Or (preferred) don't hold locks across `.await`.
- **Error type plumbing.** The first time you add a new error variant, you'll touch multiple files. It's normal; `#[from]` in thiserror makes it tolerable.
- **Build times.** Cold `cargo build` for Tauri is 2ŌĆō5 min. Incremental is fast. Consider `sccache` and `mold` linker if it hurts.
